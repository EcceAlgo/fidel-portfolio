## Introduction

Ce document présente la vision globale du projet **Fidel Portfolio**. Il définit le contexte, les objectifs, les hypothèses de travail ainsi que les orientations générales qui guideront la conception et le développement de l'application.

Il constitue le document de référence de la phase de cadrage et servira de base à la rédaction du cahier des charges, des spécifications fonctionnelles et des spécifications techniques.

## 1. Présentation du projet

Fidel Portfolio est une application web de type portfolio professionnel destinée à présenter mon profil, mes réalisations et mes compétences techniques dans les domaines du développement logiciel, du DevOps et du DevSecOps.

Le site met en valeur mon parcours, mes compétences techniques, mes réalisations, ainsi que les différents projets que j'ai développés. Il constitue un support de présentation destiné aux recruteurs, aux entreprises et à toute personne souhaitant découvrir mon travail.

Au-delà de son rôle de vitrine, ce projet est également conçu comme un démonstrateur d'ingénierie logicielle. Il constitue également un projet personnel me permettant d'expérimenter et de mettre en œuvre les bonnes pratiques de conception, de développement, de sécurité, d'automatisation, de déploiement et de supervision d'une application moderne.

## 2. Contexte

Dans le secteur du développement logiciel, un curriculum vitae ne permet pas, à lui seul, d'évaluer les compétences techniques d'un candidat ni sa manière de travailler.

Les recruteurs recherchent aujourd'hui des preuves concrètes de la capacité d'un développeur à concevoir, développer, sécuriser, tester et déployer une application. Ils souhaitent également comprendre l'organisation du projet, les choix techniques réalisés et la méthodologie adoptée.

Un portfolio technique constitue ainsi un excellent complément au CV. Il permet de présenter des réalisations concrètes, de documenter les différentes étapes d'un projet et de démontrer la maîtrise de l'ensemble du cycle de développement logiciel.

Dans le cadre d'un profil orienté DevOps et DevSecOps, il est également pertinent de mettre en évidence les pratiques liées à l'automatisation, à la sécurité, à l'intégration continue, au déploiement et à la supervision d'une application.

Les plateformes comme GitHub et LinkedIn sont devenues des éléments importants du processus de recrutement des profils techniques. Elles permettent aux recruteurs d'aller au-delà du CV en consultant directement les réalisations des candidats.

## 3. Problématique

Comment concevoir une application web permettant de présenter non seulement les réalisations d'un développeur, mais également sa démarche d'ingénierie logicielle, depuis la conception jusqu'au déploiement et à la supervision de ses projets ?

## 4. Hypothèses de travail

Les hypothèses suivantes sont retenues pour la première version du projet.

### Consultation

- Les visiteurs consulteront principalement le portfolio depuis un ordinateur.
- Une consultation sur smartphone sera également possible, notamment via un QR Code lors de salons, d'événements ou de rencontres professionnelles.
- La durée moyenne d'une visite est estimée entre deux et trois minutes.

### Recrutement

- Les recruteurs consulteront principalement le portfolio avant un entretien afin d'évaluer le profil du candidat.
- Le portfolio pourra également servir de support pendant un entretien technique.

### Projets

- Les projets présentés seront majoritairement terminés.
- Certains projets pourront être encore en évolution mais devront être suffisamment aboutis pour être présentés.
- Les projets abandonnés ne seront pas publiés.

### Maintenance

- Le portfolio sera régulièrement mis à jour afin d'intégrer les nouveaux projets, les nouvelles compétences et les évolutions professionnelles.

### Utilisateur

- La première version du projet est destinée à un utilisateur unique : son propriétaire.

## 5. Objectifs du projet

### Objectifs métier

- Valoriser mon profil.
- Renforcer ma visibilité.
- Faciliter l'évaluation de mes compétences.

### Objectifs fonctionnels

- Afficher le parcours.
- Afficher les projets.
- Afficher les expériences.
- Afficher les compétences.
- Afficher les certifications.
- Afficher les coordonnées.
- Rediriger vers GitHub.
- Rediriger vers LinkedIn.

### Objectifs techniques

Ce projet a également pour objectif de démontrer ma capacité à :

- concevoir une architecture logicielle cohérente ;
- documenter un projet selon une démarche professionnelle ;
- appliquer les bonnes pratiques de développement ;
- intégrer les principes de sécurité dès la conception ;
- automatiser les phases de validation, de test et de déploiement ;
- mettre en place une chaîne CI/CD ;
- superviser une application ;
- produire un code maintenable, évolutif et de qualité.

### Objectif d'apprentissage

Ce projet constitue également un support d'entraînement destiné à consolider mes compétences en ingénierie logicielle. Il me permet de retrouver une démarche professionnelle complète, depuis l'analyse du besoin jusqu'au déploiement et à la supervision d'une application, afin de renforcer mes acquis en développement, DevOps et DevSecOps.

## 6. Critères de réussite

### Fonctionnels

Le projet sera considéré comme réussi si :

- toutes les fonctionnalités prévues dans le MVP sont implémentées ;
- le portfolio est accessible publiquement ;
- les visiteurs peuvent consulter facilement les différentes sections du site ;
- les informations essentielles sont accessibles en quelques clics.

### Expérience utilisateur

- la navigation est intuitive ;
- les pages principales s'affichent rapidement ;
- l'interface est responsive ;
- les visiteurs identifient rapidement les informations recherchées sans assistance.

### Techniques

- le projet est entièrement documenté ;
- l'architecture est cohérente et compréhensible ;
- le code est organisé et maintenable ;
- les bonnes pratiques de développement sont respectées.
- le projet est facilement déployable.
- le projet est facilement maintenable.

### Qualité

- chaque fonctionnalité est testée avant sa mise en production ;
- les choix techniques sont documentés et justifiés ;
- la documentation permet à un développeur tiers de comprendre le fonctionnement du projet.
- la documentation est cohérente avec l'implémentation du projet.

### Sécurité

- les principales bonnes pratiques de sécurité définies pour le projet sont mises en œuvre ;
- aucune donnée sensible n'est exposée publiquement ;
- les recommandations de sécurité retenues pour le projet sont respectées.

## 7. Public cible

Le projet s'adresse principalement aux professionnels impliqués dans le recrutement de profils techniques, notamment :

- les recruteurs spécialisés dans les métiers de l'informatique ;
- les responsables techniques (Lead Developer, Engineering Manager, Responsable DevOps, etc.) ;
- les cabinets de recrutement et chasseurs de têtes.

À titre secondaire, le portfolio est également destiné :

- aux entreprises recherchant un prestataire ou un développeur indépendant ;
- aux clients souhaitant découvrir mes réalisations avant une collaboration ;
- aux développeurs et aux personnes intéressées par mon parcours et mes projets.

## 8. Proposition de valeur

Contrairement à un curriculum vitae ou à un profil LinkedIn, Fidel Portfolio ne se limite pas à présenter un parcours professionnel ou une liste de compétences.

Il offre une démonstration concrète de mes réalisations, de ma démarche d'ingénierie logicielle et de ma manière de concevoir, développer, documenter, sécuriser et déployer une application.

Le portfolio permet ainsi aux recruteurs, responsables techniques et clients potentiels d'évaluer plus facilement mes compétences à travers des preuves concrètes, des projets documentés et une présentation structurée de mon travail.

Il constitue un support vivant, régulièrement mis à jour, reflétant l'évolution de mes compétences et de mes réalisations professionnelles.

## 9. Produit Minimum Viable (MVP)

La première version du projet devra permettre :

### Présentation

- consulter une page d'accueil ;
- découvrir le profil du propriétaire du portfolio ;
- consulter les domaines de compétences.

### Réalisations

- consulter la liste des projets ;
- consulter une fiche détaillée pour chaque projet ;
- accéder aux démonstrations et aux dépôts GitHub lorsqu'ils sont disponibles.

### Contact

- accéder aux différents moyens de contact ;
- contacter le propriétaire du portfolio.

### Documentation

- consulter la documentation publique des projets.

## 10. Évolutions envisagées

Les évolutions suivantes ont été identifiées. Elles seront priorisées en fonction de leur valeur ajoutée pour le projet et de leur capacité à démontrer mes compétences.

### Priorité élevée

Ces évolutions renforcent directement la démonstration de mes compétences en ingénierie logicielle, DevOps et DevSecOps.

- Mise en place d'un système de supervision de l'application.
- Visualisation des informations relatives au déploiement continu.
- Consultation de la documentation technique directement depuis le portfolio.
- Mise en place d'un espace d'administration sécurisé permettant de gérer le contenu du portfolio.

### Priorité moyenne

Ces évolutions améliorent l'expérience utilisateur.

- Internationalisation du portfolio.
- Filtres de recherche sur les projets.
- Génération dynamique de CV.
- Statistiques de consultation.

### Priorité faible

Ces évolutions apportent un confort supplémentaire mais ne sont pas essentielles.

- Blog technique.
- Personnalisation de l'interface.

### Fonctionnalités volontairement exclues

Afin de conserver un périmètre cohérent avec les objectifs du projet, les fonctionnalités suivantes ne sont pas prévues :

- Système de messagerie interne.
- Commentaires publics.
- Forum de discussion.
- Réseau social.

## 11. Contraintes du projet

### Contraintes budgétaires

- Le coût d'hébergement de l'application devra être limité.
- L'objectif est de privilégier une solution gratuite ou dont le coût n'excède pas 5 € par mois.

### Contraintes de planning

- La première version (Produit Minimum Viable) devra être réalisée dans un délai d'un mois.

### Contraintes techniques

- Le code devra être lisible, maintenable et évolutif.
- L'application devra être responsive.
- Le projet devra être documenté.
- L'application devra fonctionner sur les principaux navigateurs modernes.

### Contraintes de sécurité

- Aucune donnée sensible ne devra être exposée.
- Les secrets devront être protégés.
- Les bonnes pratiques de sécurité devront être respectées.
- Les dépendances devront être maintenues à jour.

### Contraintes de maintenance

- Le projet devra être facilement modifiable.
- La documentation devra être maintenue à jour.
- Le déploiement devra pouvoir être reproduit facilement.

### Contraintes de performance

- Les principales pages devront se charger rapidement.
- L'application devra rester légère.
- L'application devra être utilisable avec une connexion Internet standard.

### Contraintes légales

- Le projet ne stockera pas de données personnelles.
- La prise de contact s'effectuera via le client de messagerie de l'utilisateur.

### Contraintes de qualité

- Toute décision technique devra être justifiée par un besoin identifié dans la documentation du projet.
